## 四种灰度方案

在上一节，我们用加权平均法实现了灰度转换。但那不是唯一的方案。事实上，至少有四种常见的灰度公式，每一种都有自己的出发点。

### 方案一：平均法

```glsl
float gray = (color.r + color.g + color.b) / 3.0;
```

**思路**：红绿蓝一视同仁，各取三分之一。

**优点**：简单到不需要记忆。计算量最小。

**缺点**：视觉上不准确。蓝天变得太暗，草地变得不够亮。因为人眼对绿色最敏感——给绿色和蓝色相同的权重是一种浪费。

**什么时候用它**：快速原型。当你在尝试一个新想法、不确定它能不能跑通的时候——先用平均法跑起来，验证逻辑，再替换成更好的公式。

### 方案二：加权法（ITU-R BT.601 亮度系数）

```glsl
float gray = dot(color.rgb, vec3(0.299, 0.587, 0.114));
```

**思路**：根据人眼的视觉敏感度加权——绿色最重，蓝色最轻。

**优点**：这是**视觉上最准确的**灰度转换。被广播电视行业用了半个多世纪。

**缺点**：系数不好记（谁会没事背 0.299 啊）。

**什么时候用它**：最终产物。当你的光影要给别人看的时候——用这组系数。

### 方案三：去饱和度法

```glsl
float gray = (max(color.r, max(color.g, color.b)) + min(color.r, min(color.g, color.b))) / 2.0;
```

**思路**：取最亮和最暗通道的平均值。来自 HSL 颜色模型中"亮度"的定义。

**优点**：保留了一些原图的"色彩感"——比如纯红的物体仍然偏亮，不会因为绿色权重低而变暗。

**缺点**：视觉上不够"灰"。更像是一种"褪色"而非真正的黑白。

**什么时候用它**：特效。比如你想模拟"色彩正在慢慢流失"的动画过渡——而不是真的要做黑白摄影。

### 方案四：感知均匀法（ITU-R BT.709 / sRGB）

```glsl
// 先做 Gamma 校正，再算亮度，再压回去
vec3 linearColor = pow(color.rgb, vec3(2.2));
float gray = dot(linearColor, vec3(0.2126, 0.7152, 0.0722));
gray = pow(gray, 1.0/2.2);
```

**思路**：先消除 Gamma 编码的影响，在"线性光"空间中计算亮度，再做 Gamma 编码。

**优点**：物理上最准确。

**缺点**：代码更长、计算量更大、视觉差异对大部分 Minecraft 场景来说**几乎不可见**。

**什么时候用它**：如果你在做一个追求极致准确的光影包。实际上 BSL 和大多数光影用的都是方案二的简化版。

>  如果你现在觉得"天哪，我只是想做个灰度，为什么有这么多方案"——**完全正常。** 选方案二，忘掉其他三个。等你需要它们的时候，你会想起来的。

## `dot()` 的直觉：为什么它等于"加权求和"？

我在上一节用了 `dot()` 来做加权平均。你心里可能有个小问号："点积不是用来算角度的吗？怎么突然变成加权求和了？"

这是很多人第一次学线性代数时的困惑——同一个操作有两种完全不同的解释。让我把它们说清楚。

### 解释一：几何意义

在空间里，两个向量的点积等于：

$$
\vec{a} \cdot \vec{b} = |\vec{a}| \times |\vec{b}| \times \cos\theta
$$

这个公式告诉你**两个向量"有多接近"**。如果方向相同（θ=0），点积最大；垂直（θ=90°），点积为 0；相反（θ=180°），点积为负。

这是点积的"空间直觉"——你用它来算光照（法线和光线方向的夹角）、判断物体朝向。

### 解释二：代数意义

展开来看：

$$
\vec{a} \cdot \vec{b} = a_x \times b_x + a_y \times b_y + a_z \times b_z
$$

这就是**逐分量相乘再求和**——也叫"加权求和"。如果你把 `b` 看作"权重向量"，那么点积就是在算加权平均（差一个除以总权重的步骤）。

### 在我们的代码里

```glsl
dot(color.rgb, vec3(0.299, 0.587, 0.114))
```

展开就是：

```
color.r × 0.299 + color.g × 0.587 + color.b × 0.114
```

这里我们没有用点积的"几何意义"——我们用的是它的"代数意义"。`color.rgb` 是一个三维向量（RGB 三个分量），`vec3(0.299, 0.587, 0.114)` 是另一个三维向量（三个权重）。点积帮我们一次性完成了三次乘法和两次加法。

> **点积有两种"人格"：作为几何工具，它量角度；作为代数工具，它做加权求和。在着色器编程里，两种用法都很常见。**

## `swizzling`：一个你会反复用到的小技巧

你在代码里看到了 `color.rgb` 和 `color.r`。这是 GLSL 的一个特性叫做 **swizzling**（读音像 "swizzling"——搅拌的意思）。

你可以用 `.rgba`（颜色语义）或 `.xyzw`（空间语义）来访问向量的分量：

```glsl
vec4 v = vec4(1.0, 2.0, 3.0, 4.0);

v.rgb   // vec3(1.0, 2.0, 3.0)  —— 取前三个颜色分量
v.xyzw  // vec4(1.0, 2.0, 3.0, 4.0)  —— 空间语义（和 rgba 完全等价）
v.xxx   // vec3(1.0, 1.0, 1.0)  —— 把第一个分量复制三次
v.bgr   // vec3(3.0, 2.0, 1.0)  —— 重排分量顺序
v.rg    // vec2(1.0, 2.0)  —— 只取两个
```

回到我们的灰度代码：

```glsl
color.rgb = vec3(grayscale);
```

这行的意思是：把 `color` 的 R、G、B 三个分量都设为 `grayscale`——但**保持 alpha 通道不变**。

如果你写成：

```glsl
color = vec4(grayscale);  //  错误
```

那 `color.a` 也会被覆盖——透明度信息丢失。虽然在这个简单的例子里没什么后果，但在复杂的多 pass 光影中，搞丢 alpha 信息可能导致水面消失、玻璃变不透明之类的问题。

>  **操作颜色时，始终注意你是在写 `.rgb` 还是整个 `.rgba`。** 一般情况下，保持 alpha 不变是最安全的选择。

## "先跑再优化"哲学

在着色器编程中有一个非常重要的心态——我称之为"先跑再优化"。

当你面对一个新的效果时，你的大脑里会有两种声音：

- 声音 A："先用最简单的办法让它跑起来"
- 声音 B："你必须从一开始就用最优算法，否则以后重构会很痛苦"

**听声音 A 的。**

原因很简单：着色器不像普通软件。你没法在 GPU 上打断点、没法 `printf`、没法单步调试。如果你一开始就写了一个完美的、复杂的实现——然后它不工作——你根本不知道哪个环节出了问题。

但如果你先用一个粗糙但简单的版本跑通了——比如用平均法做灰度、用硬编码的数值代替精确计算——你就有了一个"它至少能跑"的基线。然后你可以**一个一个地**替换成更好的实现，每次替换后立刻验证。

这种开发方式在图形学领域有一个专门的词，叫"**迭代式视觉验证**"（Iterative Visual Verification）——但简单来说就是：**写一点，看一眼，再改。**

## 本章要点

1.  **四种灰度方案**：平均法（快但不准）、加权法（准但不好记）、去饱和度法（更像褪色）、感知均匀法（最准确但没必要）
2.  **`dot()` 的双重身份**：几何意义（算角度）和代数意义（加权求和）——灰度用的是后者
3.  **swizzling**：`.rgb` 和 `.rgba` 的区别——大多数情况下只改 `.rgb` 保留 alpha
4.  **"先跑再优化"**：着色器编程最重要的心态——先看到效果，再追求完美

> **这里的要点是：没有人一开始就能写出完美的着色器。你现在看到的每一个炫酷的光影效果——BSL 的体积光、Complementary 的云——都是从一个粗糙的、只能勉强运行的版本迭代而来的。**
