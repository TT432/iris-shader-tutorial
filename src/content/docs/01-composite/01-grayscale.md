## 先看一眼我们面对的是什么

用 VS Code 打开 `shaders/composite.fsh`。你会看到这样一段代码：

```glsl
#version 330 compatibility

uniform sampler2D colortex0;

in vec2 texcoord;

/* RENDERTARGETS: 0 */
layout(location = 0) out vec4 color;

void main() {
    color = texture(colortex0, texcoord);
}
```

一共 10 行——去掉空行和注释，真正在做事的代码只有 7 行。但每一行都承载着特定的含义。

在开始逐行解读之前，让我先问你一个问题——**你觉得这段代码在干什么？**

你看，这几乎是一个"空操作"——它从 `colortex0` 里取了一个颜色，然后原样输出。也就是说，如果你的光影包里只有这段代码，Minecraft 的画面不会有任何变化。

那为什么它这么重要？因为它是**所有 composite 特效的骨架**。你想要灰度？在这个骨架上改一行。你想要模糊？在这个骨架上加几行。你想要光柱、Bloom、色调映射？全部都是从这 10 行开始分支的。

所以，让我们逐行拆开它。不是为了背语法——而是为了理解"为什么这样写"。

---

## 第一行：`#version 330 compatibility`

```glsl
#version 330 compatibility
```

我们在第 0.2 章已经见过这行了。它是一个**预处理器指令**——不是在 GPU 上执行的代码，而是在编译之前告诉编译器："请用 GLSL 3.30 版本的规则来理解后面的代码。"

`compatibility` 关键字：保留一些旧版 OpenGL 的功能。如果你只写 `#version 330`，Minecraft 特有的某些变量（比如 `gl_TextureMatrix`）可能会消失，导致你的着色器编译失败。

>  这里非常重要：**永远是 `#version 330 compatibility`，不要漏掉 `compatibility`。** 这是新手最容易犯的编译错误。

---

## 第二行：`uniform sampler2D colortex0;`

```glsl
uniform sampler2D colortex0;
```

这一行包含了两个你需要理解的概念：`uniform` 和 `sampler2D`。

### 什么叫 `uniform`？

回想一下我们在第 0.1 章说的：片元着色器会对**每一个像素**运行一次。总共 200 万次。

现在想象一个场景：200 万个人同时在做同一份试卷。如果试卷上写着"第 3 题的答案是 ___"，而答案对所有人来说都是一样的——比如 π = 3.14159——那么你不需要把这个数字印 200 万份。你只需要在考场前面的黑板上写一次，所有人都能看到。

`uniform` 就是这个黑板。**一个 `uniform` 变量的值在整个着色器执行期间保持不变——不管着色器被运行了多少次。** 200 万个像素看到的都是同一个 `colortex0`。

与此相对的是 `in`（我们马上会看到）——它的值**每个像素都不一样**。

### 什么叫 `sampler2D`？

`sampler2D` 是一种特殊的类型——它不是普通的数字，而是一张 **2D 纹理的"引用"** 。你可以把它理解为"一张照片的编号"。你拿着这个编号，就能随时从照片里取色。

在 Minecraft 的光影系统里，`colortex0` 不是随便一张照片——它是 **Iris 在运行你的着色器之前，已经渲染好的 Minecraft 场景画面。** 换句话说，`colortex0` = "你的着色器开始工作之前，屏幕上已经画好了的东西。"

想象一下：Iris 先把整个 Minecraft 世界画完——方块、天空、云、生物——然后把这幅画交给你："喏，这是原图。你想怎么改？"

你的 `composite.fsh` 就是这个"改图"的过程。

---

## 第三行：`in vec2 texcoord;`

```glsl
in vec2 texcoord;
```

如果 `uniform` 是"黑板上的公共信息"，`in` 就是"每个人座位上独有的试卷"。

`texcoord` 是一个二维向量，它的值**每个像素都不一样**。它告诉你：当前这个像素对应纹理上的哪个位置。

具体来说：
- `(0.0, 0.0)` = 屏幕左下角
- `(1.0, 1.0)` = 屏幕右上角
- `(0.5, 0.5)` = 屏幕正中心

它是由**顶点着色器**（`composite.vsh`）传过来的。我们会在第 1.4 节深入顶点着色器，但现在你只需要知道：`texcoord` 像一个"坐标纸条"，从顶点着色器一路传到了片元着色器，告诉每个像素"你在纹理上的位置是 (0.37, 0.82)"。

---

## 第四行：`/* RENDERTARGETS: 0 */`

```glsl
/* RENDERTARGETS: 0 */
```

这是一条**注释**——但它不是普通的注释。Iris 会专门扫描这个注释，来决定**你的着色器输出写到哪个缓冲区**。

`RENDERTARGETS: 0` 的意思是：把所有输出颜色写入 `colortex0`。

你可能会想："为什么要用注释来传递信息？这听起​​来好 hacky。"

确实是有点 hacky。这是 Minecraft 光影系统的一个历史遗留设计——因为 GLSL 本身没有"元数据"机制，开发者只能用特殊注释来传递管线配置。不管怎样，**你必须写对**——否则你的颜色可能写到错误的位置，导致画面异常。

常见的写法还有：

```glsl
/* RENDERTARGETS: 0,1,2 */  // 同时写入 colortex0,1,2（用于 G-Buffer）
/* RENDERTARGETS: 1 */      // 只写入 colortex1
```

---

## 第五行：`layout(location = 0) out vec4 color;`

```glsl
layout(location = 0) out vec4 color;
```

这一行声明了一个输出变量。`vec4` 是四维向量——四个浮点数——在这里代表 **RGBA**（红、绿、蓝、透明度）。

`layout(location = 0)` 把这个变量绑定到 `RENDERTARGETS` 的 0 号位置——也就是 `colortex0`。如果你写了 `/* RENDERTARGETS: 0,1 */` 并且有两个输出变量，第二个就应该用 `layout(location = 1)`。

> **简单来说**：`out vec4 color` = "我要输出一个颜色"，`layout(location = 0)` = "这个颜色写到 colortex0"。

---

## 第六行：`void main() {`

```glsl
void main() {
```

这行你已经在无数编程语言里见过了——程序的入口点。只不过在这里，`main()` 会被 GPU 调用 200 万次。

---

## 第七行：`color = texture(colortex0, texcoord);`

```glsl
color = texture(colortex0, texcoord);
```

**整个着色器唯一做事的代码。**

`texture()` 是 GLSL 的纹理采样函数。它从一张纹理（第一个参数）中，取出指定坐标（第二个参数）处的颜色，返回一个 `vec4`。

在这里：
- 第一个参数 `colortex0`：我们要采样哪张纹理？→ Iris 渲染好的场景画面
- 第二个参数 `texcoord`：采样哪个位置？→ 这个像素对应的坐标

所以这句话翻译成人话就是：

> **"从场景画面中，取出当前像素对应位置的颜色，复制到输出。"**

什么都没改。原样输出。这就是为什么 Base-330 是"空白模板"。

## 让它变灰：内心独白

好了。骨架已经清楚了。现在我们要给它加上一点肉——把画面变成黑白的。

闭上眼睛想一想：**什么样的颜色是"灰色"？**

如果我问你"红色是什么样的"，你会说 `(1, 0, 0)`。如果我问"白色"，你会说 `(1, 1, 1)`。

那灰色呢？灰色是——**红、绿、蓝三个通道的值相等。** 比如 `(0.5, 0.5, 0.5)` 是一种灰色，`(0.8, 0.8, 0.8)` 是浅灰，`(0.2, 0.2, 0.2)` 是深灰。

所以要把彩色变灰，我们需要做一件事：**算出当前颜色的"亮度"，然后把 R、G、B 三个通道都设为这个亮度值。**

好。问题变成了——**怎么算一个颜色的亮度？**

第一个想法：取平均值。

$$
\text{亮度} = \frac{R + G + B}{3}
$$

这是一个非常自然的想法——红色、绿色、蓝色的"分量"一样多嘛，各占 1/3。

用 GLSL 写出来就是：

```glsl
void main() {
    color = texture(colortex0, texcoord);
    float grayscale = (color.r + color.g + color.b) / 3.0;
    color.rgb = vec3(grayscale);
}
```

试试看。把它写进你的 `composite.fsh`，保存，按 F3+R 重载光影。

画面确实变灰了。但感觉……不太对？特别是蓝天——感觉变成了一种脏脏的灰色，没有你预期的那种通透感。

---

## 为什么"平均法"不够好？

这里有一个生物学的秘密：**人眼对不同颜色的敏感度不同。**

你的视网膜上有三种视锥细胞——分别对红色、绿色、蓝色敏感。但它们的数量**不相等**。对绿色敏感的细胞最多，红色次之，蓝色最少。

这意味着：同样强度的绿光和蓝光，人眼会觉得绿光**亮得多**。

如果你把 `(0, 1, 0)`（纯绿）和 `(0, 0, 1)`（纯蓝）同时用平均法变灰，它们都会变成 `0.333`——但你的眼睛会告诉你，绿色那块"明显更亮"。

所以，一个好的灰度转换应该**加权**：

$$
\text{亮度} = 0.299 \times R + 0.587 \times G + 0.114 \times B
$$

这些系数（0.299, 0.587, 0.114）不是随便编的——它们是国际电信联盟（ITU）根据人眼视觉特性制定的标准，称为"亮度系数"（luma coefficients）。

在 GLSL 里，我们可以用点积来优雅地实现：

```glsl
float grayscale = dot(color.rgb, vec3(0.299, 0.587, 0.114));
```

`dot(a, b)` 把两个向量的对应分量相乘然后相加——正好就是加权平均的数学表达。

---

## 内心独白：等一下，`dot` 怎么就算出加权平均了？

如果你看到 `dot` 这个词觉得心里没底——没关系，让我们把它拆开。

`dot(vec3(r, g, b), vec3(0.299, 0.587, 0.114))` 等价于：

```
r × 0.299 + g × 0.587 + b × 0.114
```

就是三个乘法再加起来。`dot` 只是帮你省掉了写 `r*0.299 + g*0.587 + b*0.114` 的麻烦。

> 顺便说一下：如果你觉得 "0.299, 0.587, 0.114" 太难记——可以用 `vec3(1.0/3.0)`（平均法）先跑起来，然后再替换成加权系数。**着色器编程的一大乐趣就是：你可以先看到效果，再慢慢优化。**

## 完整代码

```glsl
#version 330 compatibility

uniform sampler2D colortex0;

in vec2 texcoord;

/* RENDERTARGETS: 0 */
layout(location = 0) out vec4 color;

void main() {
    color = texture(colortex0, texcoord);
    float grayscale = dot(color.rgb, vec3(0.299, 0.587, 0.114));
    color.rgb = vec3(grayscale);
}
```

保存，F3+R。

你的 Minecraft 世界变成了黑白。蓝天变成了恰当的浅灰，草地因为绿色权重大所以偏亮——这才是"正确的"灰度。

![Minecraft 世界应用加权灰度滤镜后的效果——蓝天变为浅灰，草地因绿色权重大偏亮](/images/screenshots/ch1_1_grayscale.png)

 **恭喜。你刚刚写了一个 GPU 程序，对屏幕上 200 万个像素中的每一个独立地计算了加权平均。** 200 万次 `dot()`，同时完成。

---

## 回顾一下你学到了什么

在写这个简单的灰度滤镜的过程中，你其实已经触及了着色器编程中最核心的概念：

1. **`uniform`** = 全局常量——200 万个像素看到的是同一个值
2. **`in`** = 每个像素独有的输入——从顶点着色器一路传过来的"坐标纸条"
3. **`out`** = 输出——你的着色器"画"出的颜色
4. **`texture(sampler, coord)`** = GPU 上的取色器——从纹理中采样
5. **`dot()`** = 数学工具箱里的"加权求和器"
6. **`color.rgb` = `vec3(x)`** = 把三个通道设为同一个值

这些概念虽然简单，但它们是你之后写任何复杂特效的积木。体积光、SSAO、Bloom——所有这些"高级效果"，底层操作无非就是**采样纹理 → 做一些数学运算 → 输出颜色**。

> **这里的要点是：一个着色器就是三件事——从纹理中取色、做数学运算、把结果写回去。灰度滤镜如此，体积光也如此。**
