好吧，我们开始吧。

---

## 一个问题

想象你在一个巨大的仓库里。仓库里有 200 万个灯泡——排成 1920 列、1080 行，正好是一块屏幕的大小。你的任务是：**在一秒钟之内，把每一个灯泡的颜色设置好，拼成一幅 Minecraft 的画面。**

你会怎么做？

你当然可以自己一个人跑遍整个仓库，一个一个拧灯泡。从左上角开始，第一行，第一列——绿色。第一行，第二列——棕色。第一行，第三列——蓝色……等你拧完 200 万个灯泡，可能已经过去了好几天。Minecraft 早就不是 Minecraft 了，它变成了一幅幻灯片——而且是那种每张幻灯片要等三天的幻灯片。

但你的电脑**每秒钟**要更新 60 次整个屏幕。60 × 200 万 = 1.2 亿个像素/秒。一个人？不可能。

**所以计算机不派一个人去。它派了一支军队。**

---

## 大脑与千手观音

好吧，"军队"这个比喻还不够好。让我换一种说法。

CPU 像一位数学教授。他戴着一副厚厚的眼镜，坐在一间安静的办公室里。你给他一道题——比如说，算 `sin(3.14159) × log(42.7)`——他眨眼之间就能给你精确到小数点后 15 位的答案。他非常聪明，能做复杂的分支判断："如果 x > 0 且 y < 0 且 z 是质数，那么……"——他处理这些逻辑游刃有余。

但他有一个致命弱点：**他一次只能做一件事。** 如果你给他 100 道题，他得一道一道做。第 1 题→第 2 题→第 3 题……

GPU 呢？GPU 根本不聪明。如果你让 GPU 单独算 `sin(3.14159) × log(42.7)`，它可能比教授慢得多。但 GPU 有一项教授望尘莫及的能力——它有一千只手。

想象一尊千手观音。每一只手都很笨——只会做简单的加减乘除。但如果你对着一千只手同时喊："所有人！拿起红色灯泡！拧上去！"——它们会**同时**完成。从教授的角度看，这简直像魔法：一秒钟，200 万个灯泡全部换好。

这就是 GPU 的核心秘密：**它不是计算得快，它是同时计算的数量多。** 用行话说，这叫做"大规模并行"（Massively Parallel）。

![CPU 串行思考 vs GPU 并行洪流——着色器的核心思想：把同一段程序发给很多数据](/images/cpu_vs_gpu.png)

等等——你可能会说："这不就是分工合作吗？"

不是的。如果是分工，200 万个灯泡会分成 1000 组，每组 2000 个，然后每只手按顺序处理自己那组的 2000 个——这依然需要等待。

GPU 的更极端：它给了**每一个灯泡一只专属的手**。200 万个灯泡 = 200 万只手。所有手同时动作。没有排队，没有"该你了"。

> **并行不是把工作分给多个人。并行是给每个工作单元配一个专属执行者。**

---

## 这和着色器有什么关系？

关系大了。

当你在 Minecraft 里安装一个光影包，你写的着色器代码会被 GPU 执行。但关键问题是——**它会被执行多少次？**

不是一次。不是一千次。是**屏幕上的每一个像素，一次。**

如果 Minecraft 窗口是 1920×1080，那么你写的每一行片元着色器代码，都会被 GPU 同时运行 2,073,600 次。每个像素独立地运行同一段代码，算出自己的颜色。

这个事实如此重要、如此反直觉，以至于我要再写一遍：

> **你写的着色器不是一个"处理所有像素的程序"。你写的是一个"处理单个像素的函数"。GPU 负责同时运行它几百万份。**

---

## 一个具体的例子

假设你想要整个屏幕变成红色。在 CPU 编程中，你可能会这样写：

```python
# CPU 的做法：循环遍历每个像素
for y in range(1080):
    for x in range(1920):
        screen[y][x] = (255, 0, 0)  # 红色
```

但在 GLSL（GPU 着色语言）中，你**不能写循环遍历像素**。相反，你只写一个像素的处理逻辑：

```glsl
void main() {
    // 这段代码被 GPU 同时运行 200 万次
    // 每次运行时，它只负责一个像素
    fragColor = vec4(1.0, 0.0, 0.0, 1.0); // 红色
}
```

看出区别了吗？在 CPU 版本里，你明确地告诉计算机："从 (0,0) 开始，往右走，到头后换下一行"——这是一个**指令序列**。而在 GPU 版本里，你只是在说："不管你被分配到哪个像素，把它涂红。"——至于谁分配到哪个像素、以什么顺序执行，**你管不着**。GPU 自己决定。

这就是"着色器思维"和"循环思维"的根本区别。你失去了控制执行顺序的能力，但你获得了让 200 万个像素同时计算的能力。

用一句行话来说：**着色器是无状态的。** 每次调用 `main()` 不记得上一次调用时发生了什么——也不应该记得。

---

## 每个像素是孤岛

这里有一个非常微妙但至关重要的推论：**每个像素的着色器执行是完全独立的。**

当 GPU 同时运行你的 `main()` 函数 200 万份时，每一份只能看到**自己的像素**。左边的像素不知道右边像素的颜色。上面的像素不知道下面像素在干什么。

你可能会想——"那我怎么做模糊效果？模糊需要看周围像素的颜色啊！"

好问题。答案是：你确实不能直接在着色器里"问"隔壁像素的颜色。但你可以通过一些间接手段——比如多次渲染（pass）、或者从上一帧保存的纹理中读取——来模拟这种效果。这是着色器编程中最核心的技巧之一，我们在后面的章节中会花大量时间讨论。现在，你只需要记住这个限制：

>  **着色器实例之间互相隔离。** 一个像素不能直接读取另一个像素的结果。要"看到"邻居，你必须借助纹理——把上一帧的画面存下来，在下一帧读取。

---

## 那么顶点着色器呢？

到目前为止，我们讨论的都是片元着色器（fragment shader）——处理像素的那个。但一个完整的光影包还有另一种着色器：顶点着色器（vertex shader）。

片元着色器处理的是**像素**，而顶点着色器处理的是**顶点**。

Minecraft 的世界是由方块组成的。每个方块有 8 个顶点（立方体的 8 个角）。如果你的视野里有 10,000 个方块，那就是 80,000 个顶点需要处理。顶点着色器的任务就是计算每个顶点**在屏幕上的最终位置**。

它同样遵循并行原则——每个顶点独立执行，不知道其他顶点的命运。

```glsl
// 顶点着色器：每个顶点运行一次
void main() {
    // 把顶点从"3D 世界"变换到"2D 屏幕"
    gl_Position = gl_ModelViewProjectionMatrix * gl_Vertex;
}
```

这段代码被同时运行了几万次——每个顶点一次。`gl_Vertex` 是当前顶点的 3D 坐标，`gl_ModelViewProjectionMatrix` 是一个矩阵，它把 3D 坐标转换成屏幕上的 2D 坐标。

矩阵乘法听起来吓人，但你暂时不需要理解它。现在你只需要记住分工——**顶点着色器负责把东西放到屏幕上的正确位置，片元着色器负责给每个位置涂上正确的颜色。** 前者搭骨架，后者上色。我们在第 2 章会深入顶点着色器。

## 一个初学者特别容易犯的错误

很多第一次接触着色器的人会想："既然片元着色器每个像素运行一次，那我可以根据像素的位置来画不同的东西，对吧？前 100 行画天空，中间画地面，底下画石头？"

**不对！你做不到！**

在片元着色器里，你**不知道你正在处理的像素是天空、地面、还是石头。** 你只有一个纹理坐标和一个屏幕坐标。这些信息告诉你像素在屏幕上的位置，但**不告诉你这个位置对应的是游戏世界里的什么东西。**

要区分天空和地面，你需要借助其他缓冲区——比如深度缓冲（告诉你这个像素离相机有多远）、法线缓冲（告诉你这个像素对应的表面朝向）——来推断每个像素代表什么。

简单来说：**片元着色器是一个盲人画家。** 它的画布上有 200 万个格子，但它不知道哪个格子是天空、哪个是草方块。它只能根据你给它的"线索"——纹理坐标、深度值、法线方向——来猜测自己应该涂什么颜色。

提供这些线索，就是 G-Buffer（几何缓冲区）的职责——我们在第 2 章会详细讨论。

## 本章要点

在我们继续之前，确保你同意以下几点：

1.  **CPU 是数学教授**：聪明但一次只能做一件事
2.  **GPU 是千手观音**：每只手很笨但数量极多，所有手同时动作
3.  **着色器是单像素/单顶点的处理函数**——被 GPU 并行执行数十万次
4.  **着色器实例之间互相独立**——不能直接读写其他像素的结果
5.  **顶点着色器定位，片元着色器上色**——分工明确，互不越界
6.  **片元着色器是盲人画家**——它不知道自己在画天空还是草地，只能靠"线索"推测

如果你对其中某一点还有疑问——特别是第 3 点，关于"你为什么不能控制执行顺序"——**请回头再读一遍。** 这个思维转变是整个教程的地基。地基不稳，后面会越学越困惑。这不是你的问题——这是每一个从 CPU 编程转向 GPU 编程的人都要经历的阵痛。

顺便说一下，我在学这些的时候犯过一个特别蠢的错误：我试图在着色器里写一个全局计数器来统计"已经处理了多少像素"。结果当然是——根本编译不过。因为 GPU 上根本没有"全局变量"这个概念。每个着色器实例都在自己的小宇宙里，看不到别人。

如果这些概念在你脑海里已经开始成形了，那么恭喜——**你已经推开了 GPU 编程的大门。** 剩下的路，我们一步一步走。

> **要点：CPU 是你指挥一个人做复杂的事。GPU 是你指挥一百万个傻子，每人做一件极简单的事——但一百万人同时做。**
